# Perception Pipeline: Preprocessing, PointPillars & MonoBEV

This document provides an in-depth overview of the data preprocessing routines and the two BEV encoder architectures used within the stereo-LiDAR perception pipeline: the **LiDAR branch** (PointPillars) and the **Camera branch** (MonoBEV / Lift-Splat-Shoot).

## 1. Preprocessing (`preprocessing.py`)

The preprocessing module converts raw multimodal sensor data (LiDAR point clouds and stereo camera images) into structured, computation-ready formats.

### 1.1. Bird's-Eye-View (BEV)
- **LiDAR BEV:** Projects 3D LiDAR point clouds onto a 2D regular grid, generating density and height maps from a top-down perspective. Implemented via the PointPillars branch (see Section 2).
- **Camera BEV (MonoBEV):** Lifts monocular RGB features into 3D via learned depth distributions and voxel-pools them onto the same BEV grid. Implemented via the MonoBEV / Lift-Splat-Shoot branch (see Section 3).

### 1.2. Camera Frustum Extraction
- **Description:** Extracts LiDAR returns that fall inside a 2D bounding box generated by a camera detection system.
- **Workflow:** 
  1. Points are transformed from the ego-frame into the camera frame.
  2. Points are projected onto the image plane.
  3. Points falling inside the target 2D bounding box are retained.
  4. Selected points are transformed into a local frustum frame (centroid-aligned to the Z-axis).
  5. RGB values from the corresponding image pixels can optionally be appended to the point features.

### 1.3. Voxel Grid Representation
- **Description:** Discretizes the 3D space into a regular volumetric grid (voxels).
- **Channels Included:**
  - `Occupancy:` Binary flag indicating if the voxel contains points.
  - `Mean Height:` Normalized Z-coordinate of the points within the voxel.
  - `Point Density:` Log-normalized count of points inside the voxel.
  - `Mean Intensity:` Average reflectance/intensity of the LiDAR returns.

### 1.4. DBSCAN Clustering
- **Description:** A spatial clustering approach based on Euclidean distance to segment the point cloud into individual objects or clusters.
- **Implementation Details:** Uses a KD-Tree accelerated DBSCAN algorithm. It returns per-point labels and statistics for each cluster, including its centroid, number of points, and axis-aligned bounding box (AABB) extents.

## 2. PointPillars Architecture (`pointpillars.py`)

The LiDAR branch leverages the PointPillars architecture, a fast encoder designed to process raw point clouds into dense BEV feature maps for accurate 3D object detection.

### 2.1. Pillarization (`pillarize`)
- **Grid Setup:** The continuous 3D space is discretized in the X and Y dimensions, creating vertical columns called "pillars" with infinite height (no Z-axis binning).
- **Binning:** Points are assigned to their respective pillars. Empty pillars are ignored, creating a sparse representation.
- **Decoration:** Each raw point `[x, y, z, reflectance]` is augmented to a 9-dimensional vector containing:
  - Original features `(x, y, z, r)`
  - Offsets to the arithmetic mean of all points in the pillar `(x_c, y_c, z_c)`
  - Offsets to the geometric center of the pillar cell in the X-Y grid `(x_p, y_p)`

### 2.2. Pillar Feature Net (PFN)
- **Architecture:** A simplified version of PointNet.
- **Process:**
  - A shared Multi-Layer Perceptron (MLP) consisting of a Linear layer, BatchNorm, and ReLU is applied individually to each point.
  - A Max-Pooling operation aggregates the features across all points within the same pillar.
- **Output:** A single robust feature vector of dimension `C` (e.g., 64) represents each non-empty pillar.

### 2.3. PointPillars Scatter
- **Process:** The dense feature vectors generated by the PFN are scattered back onto their original coordinates in the 2D grid (`nx` by `ny`).
- **Output:** A dense 2D pseudo-image of shape `[C, nx, ny]`.

### 2.4. BEV Backbone 2D
- **Description:** A lightweight 2D Convolutional Neural Network (CNN) applied to the pseudo-image.
- **Purpose:** It aggregates spatial context from neighboring pillars and refines the features.
- **Output:** The final dense BEV feature map (e.g., of shape `[128, nx, ny]`), which is ready to be fused with the Camera BEV branch or passed directly to a detection head.

## 3. MonoBEV — Lift-Splat-Shoot Camera Branch (`monobev.py` + `preprocessing.py`)

The camera branch converts a single calibrated RGB image into a BEV feature map using the **Lift-Splat-Shoot (LSS)** paradigm (Philion & Fidler, NeurIPS 2020). The output grid is pixel-aligned with the LiDAR PointPillars BEV, enabling direct channel-wise fusion.

### 3.1. Configuration (`MonoBEVConfig`)

All hyperparameters live in one dataclass so callers never hard-code magic numbers:

| Field | Default | Description |
|---|---|---|
| `x_range` | `(0, 50)` m | Forward extent of the BEV grid |
| `y_range` | `(-20, 20)` m | Lateral extent |
| `bev_res_m` | `0.25` m | Metres per BEV cell |
| `d_min / d_max` | `1 / 60` m | Depth bin range |
| `num_depth_bins` | `41` | Number of depth bins D (matches LSS paper) |
| `log_depth` | `False` | Linear spacing; set `True` for log-spaced bins |
| `img_backbone_out` | `256` | Channels from the shared backbone |
| `context_channels` | `64` | C — semantic feature dimension |

The `x_range`, `y_range`, and `bev_res_m` must match `PillarConfig` so both BEV maps are spatially aligned.

### 3.2. Shared CNN Backbone (`_EfficientNetBackbone`)

A lightweight EfficientNet-inspired CNN with three stride-2 blocks (stem → block1 → block2 → 1×1 projection), downsampling the input image by 8× and producing a feature map of shape `(B, 256, H/8, W/8)`. Uses SiLU activations and BatchNorm throughout.


### 3.3. Depth Head & Context Head

Both heads branch from the shared backbone features:

- **Depth head:** `Conv2d(256→256, 3×3) → BN → ReLU → Conv2d(256→D, 1×1)`  
  Output `(B, D, H', W')` → `softmax(dim=1)` → depth probability distribution over D bins per pixel.
- **Context head:** `Conv2d(256→256, 3×3) → BN → ReLU → Conv2d(256→C, 1×1)`  
  Output `(B, C, H', W')` → per-pixel semantic feature vectors.

### 3.4. Outer Product — Building the Frustum

For each of the `H'×W'` pixel locations the module forms the **outer product** of its context vector `(C,)` and its depth distribution `(D,)`:

```
frustum_feats = context.unsqueeze(2) * depth_dist.unsqueeze(1)
             # (B, C, 1, H', W') × (B, 1, D, H', W') → (B, C, D, H', W')
```

This gives every depth bin the full context feature, scaled by its predicted probability. Bins the model is confident about carry most of the signal; uncertain bins contribute little. The result is a **frustum** of `H'×W'×D` points, each with a `C`-dim feature.

### 3.5. Lift to 3D — Unprojection to Ego Frame

Each frustum point `(u, v, d)` is unprojected to ego/vehicle coordinates:

```
ray_cam = K⁻¹ · [u, v, 1]ᵀ          # unit ray in camera frame
X_cam   = d · ray_cam                # scale by depth
X_ego   = R · X_cam + t             # apply cam→ego extrinsic
```

`K⁻¹` is computed with `np.linalg.inv` on the CPU (LAPACK, always available) to avoid loading `libtorch_cuda_linalg.so`, which requires a cuSolver version that may not match the system installation.

### 3.6. Splat — Efficient Voxel Pooling

The expensive step: scatter all `H'×W'×D` ego-frame features into BEV cells. A naïve loop is too slow. The implementation uses the **sort + cumulative-sum boundary** trick from the original LSS repository:

1. Compute flat BEV cell index `ix·ny + iy` for every frustum point; discard out-of-bounds.
2. Append the batch index to form a global sort key `batch·(nx·ny) + cell`.
3. **Sort** all valid points by this key in one `torch.argsort` call.
4. Prefix-sum the feature tensor with `cumsum(dim=0)`.
5. Detect group boundaries (where the key changes) and read off `cumsum[end] - cumsum[start]` — this gives the **sum-pool** for every non-empty cell in O(M log M) time with zero Python loops and no `scatter_add`.

```python
# Core of the cumsum trick
feats_pad = torch.cat([zeros(1, C), feats_sorted])   # (M+1, C)
cumsum    = feats_pad.cumsum(dim=0)                   # prefix sums
boundaries = where(key[1:] != key[:-1])               # group starts
group_sums = cumsum[ends] - cumsum[boundaries]        # per-cell sum
```

### 3.7. BEV Refinement

The splatted map is passed through `BEVBackbone2D` (the same lightweight 2D CNN used in the LiDAR branch) to add spatial context between neighbouring BEV cells.

**Final output:** `(B, C, nx, ny)` — aligned with the PointPillars BEV, ready for channel-wise concatenation and downstream fusion.

### 3.8. Expected Outputs Before / After Training

| State | Depth distribution | BEV appearance |
|---|---|---|
| **Untrained** | Uniform (~1/D per bin) | Triangular frustum footprint; visible depth-bin stripes; random feature values |
| **Trained** | Peaked at true object depths | Features concentrated at object locations; no stripes; BEV semantically meaningful |

The triangular frustum shape and the Y-axis centering at y=0 m are a geometry sanity-check that is visible even in the untrained model.
